## Summary
`robot src_v2` keeps ownership of robot runtime concerns (embedded webserver + embedded NATS) while using autoswap manifests for artifact/version composition.

Core intent:
1. Keep robot UI behavior nearly identical to the current generation UI (the requested `src_v3` look/flow target).
2. Keep `/natsws` on the `robot src_v2` embedded webserver.
3. Let autoswap manifests define and sync robot UI + runtime companion artifacts (`camera`, `mavlink`, and `repl`).

`src_v2` is the first robot version that formalizes artifact composition through manifest-driven autoswap while preserving robot-owned runtime endpoints.

## Scope
In scope:
- Define `robot/src_v2` as the runtime host for webserver + embedded NATS + robot endpoints.
- Define autoswap manifest composition for robot UI/runtime artifacts.
- Define plugin boundaries for camera/mavlink data producers consumed by robot UI.
- Define acceptance tests and rollout gates.

Out of scope:
- Full removal of `robot/src_v1` until parity and rollout checks pass.
- UI redesign beyond parity-level updates.

## Target Architecture
- `robot src_v2` process owns:
  - Embedded NATS lifecycle.
  - Embedded HTTP server lifecycle.
  - WebSocket endpoint at `/natsws`.
  - Robot-specific endpoints (for example camera and init/health APIs).
  - UI static serving from synced `src_v2/ui/dist`.
- `repl/src_v1` remains focused on REPL command bus/tasks; it is not the robot web endpoint owner.
- `camera` plugin binary provides camera stream data/service.
- `mavlink` plugin binary provides MAVLink ingest + publish to NATS subjects.

Data flow:
1. Browser loads robot UI from robot-hosted static mount.
2. Browser connects to `ws(s)://<host>/natsws` (same origin).
3. Robot server bridges/exposes NATS WebSocket endpoint.
4. Camera/mavlink producers publish to NATS subjects.
5. Robot UI subscribes over `/natsws` and renders state.

## Public Interfaces / Contract Changes

### 1) Robot HTTP surface (new/expanded in `src_v2`)
Required routes on robot embedded webserver:
- `GET /health` -> `ok`
- `GET /natsws` -> WebSocket upgrade endpoint for embedded NATS WS
- `GET /` + static routes served from `src_v2/ui/dist`
- `GET /api/init` compatibility payload used by robot UI bootstrap
- `GET /stream` and other robot-special endpoints required by camera/video flows

Constraints:
- `/natsws` must be same-origin, no cross-origin dependency required for default UI.
- Robot endpoints must remain compatible with existing UI behavior.

### 2) Robot manifest contract (new)
Introduce a manifest record consumed by autoswap/runtime composition.

Proposed minimal fields:
- `name`: `robot`
- `version`: `src_v2`
- `ui.mount_path`: `/`
- `ui.dist_path`: resolved artifact path (expected `src/plugins/robot/src_v2/ui/dist`)
- `ui.index`: `index.html`
- `requires`: list of runtime capabilities (`natsws`, `camera`, `mavlink`)
- `subjects`: expected NATS subjects (read/write)
  - `artifacts.sync`: explicit artifact set autoswap must place locally before activate:
  - autoswap supervisor binary (`dialtone_autoswap_v1` versioned artifact)
  - robot runtime binary (`dialtone_robot_v2` versioned artifact)
  - repl runtime binary (`dialtone_repl_v1` versioned artifact)
  - robot UI directory (`src_v2/ui/dist`)
  - camera binary (`dialtone_camera_v1` versioned artifact)
  - mavlink binary (`dialtone_mavlink_v1` versioned artifact)
  - wlan driver workflow bundle (`src/plugins/wlan/src_v1`)

Behavior:
- Robot runtime uses manifest-pinned artifacts for startup composition.
- Unknown/missing capability marks robot module unhealthy in status output.
- Autoswap downloads/syncs manifest artifacts first, including the full `src_v2/ui/dist` folder, then flips to the new active set.

### 2a) Canonical Build Output Paths (all binaries)
All build pipelines must emit into `<repo_root>/bin` so autoswap and manifests use one consistent location.

Required five binaries:
1. `<repo_root>/bin/dialtone_autoswap_v1`
2. `<repo_root>/bin/dialtone_robot_v2`
3. `<repo_root>/bin/dialtone_repl_v1`
4. `<repo_root>/bin/dialtone_camera_v1`
5. `<repo_root>/bin/dialtone_mavlink_v1`

Required UI artifact directory:
1. `<repo_root>/src/plugins/robot/src_v2/ui/dist`

Required WLAN workflow artifact directory:
1. `<repo_root>/src/plugins/wlan/src_v1`

### 3) NATS subject contract (lock for v2)
Reserve and document stable subjects for robot UI integration:
- `mavlink.>` telemetry/events
- `camera.>` stream/control metadata
- `robot.ui.>` optional UI-origin control/events

### 4) Backward compatibility
- Keep `robot/src_v1` runnable during transition.
- Keep robot UI expecting `/natsws` and avoid hardcoding alternate hostnames/ports.

## Testing and Acceptance

### Unit tests
- Robot `/natsws` handler: upgrade success/failure, upstream unavailable handling.
- Manifest parser/validator for robot module entries.
- Static `ui/dist` mount resolver path safety and fallback behavior.
- Robot special endpoints (`/api/init`, `/stream`) compatibility checks.
- Test harness implementation must use `src/plugins/test/src_v1` registry + `StepContext` APIs.
- Robot test suite structure must follow `src/plugins/test/README.md`:
  - `src/plugins/robot/src_v2/test/cmd/main.go` single orchestrator process.
  - numbered `NN_name` test folders (for example `01_smoke`, `02_runtime`, `03_ui`), each exporting registration helpers.
  - no plugin-local test context type; use `testv1.StepContext`.
- Every `testv1.Step` must define an explicit `Timeout` appropriate to the step (target: seconds, not minutes, for normal smoke/integration steps).
- Step assertions must default to action-then-wait patterns:
  - `WaitForStepMessageAfterAction`
  - `WaitForMessageAfterAction`
  - `WaitForBrowserMessageAfterAction` (browser steps)
  Avoid ad-hoc sleeps except where a protocol/driver forces a bounded wait.
- Test step logs must use `StepContext` logging methods only (`Infof/Warnf/Errorf/Debugf`) and never `fmt.Print*` direct output.

### Integration tests
- Start robot runtime with embedded NATS + HTTP + robot manifest artifacts.
- Verify:
  - `GET /health` returns 200 `ok`
  - `GET /` serves robot UI index
  - `/natsws` upgrades and can pub/sub test messages
- Simulate missing camera/mavlink capability and assert degraded status without crash.
- Integration tests must emit/assert via NATS-backed step topics so pass/fail evidence is observable through `logs` plugin streams.

### End-to-end checks
- Browser loads `rover-1` UI through robot-hosted route.
- UI receives live MAVLink telemetry over `/natsws`.
- UI receives camera state/stream metadata over `/natsws`.
- Existing tunnel target (`rover-1.dialtone.earth`) remains functional.

Mandatory final UI E2E sequence:
1. Local mock E2E:
   - Run robot UI locally against a mock data source + mock `/natsws` publisher.
   - Verify core UI flows render and react to synthetic camera/mavlink frames.
2. Hybrid local+remote E2E:
   - Run UI locally, but connect `/natsws` to remote robot endpoint.
   - Verify UI updates with live remote data over `/natsws`.
3. Capture artifacts for both:
   - test logs
   - screenshots/video where available
   - pass/fail summary linked from `src_v2/test`.
4. E2E browser steps must use `StepContext` browser APIs and wait helpers from `testv1` (aria wait/click + console/NATS waits), with explicit per-step timeouts.

## Risks and Mitigations
- Risk: UI depends on endpoints not provided by robot `src_v2` host.
  - Mitigation: explicit endpoint parity checklist and compatibility shims before cutover.
- Risk: websocket proxy regressions under unstable links.
  - Mitigation: reconnection tests + health signals + bounded retry.
- Risk: composition mismatch across plugin versions.
  - Mitigation: manifest schema validation + startup hard-fail on incompatible set.

## Defaults Chosen
- Runtime web host for robot UI: robot `src_v2` embedded webserver.
- Browser NATS path: `/natsws` (same-origin).
- Robot UI mount path: `/`.
- Transition strategy: dual-run with `src_v1` fallback until acceptance criteria pass.
