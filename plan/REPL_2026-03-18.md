## REPL Plan: 2026-03-18

### Better Direction

The first draft focused mostly on subtone lifecycle. That is still part of the problem, but it is not the main fix for keeping REPL alive in the background.

The bigger issue is architectural:

- `./dialtone.sh` currently calls `EnsureLeaderRunning(...)`
- `EnsureLeaderRunning(...)` autostarts a raw detached leader process:

```text
repl src_v3 leader --embedded-nats ...
```

- that leader owns both:
  - the REPL command loop
  - the embedded NATS broker

This means the normal CLI path depends on a single detached process that is:

- not supervised
- not restarted by policy
- not clearly versioned
- not clearly separated from stale pid detection

The codebase already has a stronger runtime model in `repl src_v3 service --mode run`, but the default `./dialtone.sh` path is not using it.

### What Happened Today

While verifying `robot src_v2` against the rover, the default `./dialtone.sh` path stopped behaving like a clean foreground proxy. A long-lived `cad src_v1 dev` subtone was still registered as active in the REPL leader, and new foreground commands like:

```bash
./dialtone.sh ssh src_v1 resolve --host rover
```

entered the routed path but did not complete cleanly in the shell.

The immediate recovery was still:

```bash
./dialtone.sh repl src_v3 process-clean
```

That killed the background leader and bootstrap helpers, which let the next command autostart a fresh leader.

### Why We Needed To Restart The Leader

The restart was not needed because the REPL code changed. It was needed because the current leader runtime does not manage long-lived foreground subtone state well enough.

Observed reasons:

- A long-lived foreground subtone (`cad src_v1 dev`) remained active in the leader registry.
- There is no normal operator command to stop one specific active subtone from the index room.
- The shell path does not clearly explain that a stale active subtone is blocking or delaying later foreground commands.
- `process-clean` is currently the only reliable recovery path when the leader state gets wedged around foreground work.

That explanation is only partially complete.

The actual problem is:

- stale subtone lifecycle control
- plus a weak background leader model

Even if subtone control improves, a detached raw leader process is still the wrong default background runtime.

### Desired Behavior

If REPL code has not changed, the same leader should stay up in the background and continue serving:

- `./dialtone.sh ssh ...`
- `./dialtone.sh robot ...`
- `./dialtone.sh autoswap ...`
- `./dialtone.sh cad ...`
- `./dialtone.sh chrome ...`

without needing a leader restart.

Restarting the leader worker should be reserved for:

- explicit operator cleanup
- binary/code updates to the leader
- unrecoverable health failure
- service-managed worker replacement

The background runtime should behave like this:

1. A single local REPL supervisor is installed or autostarted once.
2. That supervisor owns the leader worker.
3. `./dialtone.sh <plugin> ...` does not start ad hoc leaders. It only:
   - checks supervisor health
   - checks leader health via NATS
   - proxies the command
4. If the worker is unhealthy, the supervisor restarts it.
5. If the binary version changed, the supervisor replaces it.

### Root Gaps

1. Wrong default autostart target

The current autostart path starts a raw leader process instead of the service/supervisor runtime that already exists in `repl src_v3 service`.

2. Leader and embedded NATS are tied to one detached process

That is acceptable for a local dev runtime, but only if a supervisor owns it. Right now the detached process owns both and can disappear without a proper recovery story.

3. Running/healthy detection is split and misleading

We saw states like:

- `nats endpoint ... reachable=false`
- `repl leader process running=true`

That means we can detect some process that looks like a leader, while the actual NATS/health path is dead.

The health check is better now than it used to be, but the operational decision is still too weak:

- raw leader process exists
- therefore maybe we are fine

What matters is:

- supervisor healthy
- leader health subject responds
- command queue subject responds

4. No single-subtone lifecycle control

There is `subtone-list` and `subtone-log`, but no first-class:

- `subtone-stop --pid <pid>`
- `subtone-kill --pid <pid>`
- `subtone-bg --pid <pid>`

5. Foreground/background state is too weak

The leader knows whether a command was background at spawn time, but the registry does not persist that mode. So the operator cannot clearly see which active subtone is intentionally long-lived.

6. Recovery is too coarse

`process-clean` kills the whole leader stack. That works, but it is too destructive for everyday development.

7. Shell feedback is incomplete

When a new command is delayed by an active subtone, `DIALTONE>` should say so explicitly instead of looking hung.

### Recommended Changes

#### 1. Make the supervisor the default background runtime

Change the default `./dialtone.sh` proxy model from:

```text
EnsureLeaderRunning -> start detached raw leader
```

to:

```text
EnsureREPLRuntime -> ensure local repl service/supervisor is running
                 -> verify leader health over NATS
                 -> proxy command
```

This is the most important change.

Concretely:

- reuse `repl src_v3 service --mode run`
- add a local autostart wrapper for it
- stop spawning detached raw `src_v3 leader --embedded-nats` as the default background path

#### 2. Separate service health from leader health

Track at least two layers:

- supervisor health
- leader health

The supervisor should be responsible for restarting the leader worker when:

- NATS health subject stops responding
- the worker exits
- the worker binary changes

`./dialtone.sh` should not have to guess whether to start a new leader process.

#### 3. Make `leader.json` describe the whole runtime, not just the worker

Persist:

- supervisor pid
- worker pid
- nats url
- room
- version
- started_at
- last_healthy_at
- worker_started_at
- worker_last_healthy_at
- autostart mode (`service`, `manual`, `foreground`)

Then `status` can explain what is actually alive.

#### 4. Add per-subtone stop/kill

Add REPL commands:

```bash
./dialtone.sh repl src_v3 subtone-stop --pid <pid>
./dialtone.sh repl src_v3 subtone-kill --pid <pid>
```

Minimum behavior:

- `subtone-stop` sends interrupt/termination to the owning process tree
- registry entry transitions to stopped/exited
- index room emits a lifecycle summary

#### 5. Persist mode in the subtone registry

Track whether each subtone is:

- `foreground`
- `background`
- `service`

This should be visible in `subtone-list`.

#### 6. Make foreground admission explicit

When a foreground subtone is already active, the leader should do one of these clearly:

- reject a second foreground command with a policy error, or
- queue it and print a clear `DIALTONE>` message naming the blocking pid/command

Either is better than silent waiting.

#### 7. Allow promoting long-lived tasks to background/service mode

For commands like:

- `cad src_v1 dev`
- `cad src_v1 serve`
- `chrome src_v3 watch`
- tunnel helpers

the leader should have a standard model for long-lived work:

- start as background intentionally, or
- promote an active subtone to background/service mode

That keeps the index room usable for later foreground commands.

#### 8. Keep worker restart tied to health/version changes

The supervisor should only replace the worker when:

- the leader binary version changed
- the configured NATS endpoint changed
- the running leader fails a real health check

It should not restart just because some plugin work was long-lived.

#### 9. Add a doctor/status explanation for blocked foreground work

`./dialtone.sh repl src_v3 status` or a new `doctor` command should say things like:

- `foreground subtone active: pid 590839 cad src_v1 dev --port 3012 --backend-port 8082`
- `new foreground commands will queue behind this process`

### Concrete Next Steps

#### Phase 1: Fix the background runtime model

1. Replace `EnsureLeaderRunning(...)` with `EnsureREPLRuntime(...)`.
2. Make that path start or reconnect to the local `repl src_v3 service` supervisor.
3. Have the supervisor own the worker leader lifecycle.
4. Extend `leader.json` to include supervisor + worker state.
5. Make `repl src_v3 status` show:
   - supervisor pid/state
   - worker pid/state
   - NATS health
   - command subject health

#### Phase 2: Fix normal recovery without restart

6. Add `subtone-stop` and `subtone-kill`.
7. Extend the registry to store `mode=foreground|background|service`.
8. Show mode in `subtone-list`.
9. Emit explicit blocked/queued foreground messages in the shell relay.
10. Add a `doctor` summary for stale active subtones.

#### Phase 3: Improve long-lived workflows

11. Add a standard background/service model for:
    - `cad src_v1 dev`
    - `cad src_v1 serve`
    - chrome watchers
    - tunnel helpers
12. Let those commands either:
    - start directly as background/service, or
    - be promoted from foreground to background

### Recommended Policy

For normal local development:

- one background REPL supervisor per machine/user
- one leader worker under that supervisor
- all `./dialtone.sh` commands proxy through it

For the CLI:

- never autostart an ad hoc raw leader if a supervisor model is available
- if the supervisor is healthy, just proxy
- if the worker is dead, let the supervisor repair it
- if the supervisor is absent, start it once and reuse it

### Short Version

We restarted the leader today because the current default background runtime is the wrong thing: a detached raw leader process instead of a supervised REPL runtime.

The better fix is:

- make the REPL service/supervisor the default background runtime
- let it own worker restarts
- keep `./dialtone.sh` as a thin proxy
- add subtone lifecycle controls so long-lived work does not wedge the index room
